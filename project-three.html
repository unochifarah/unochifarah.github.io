Case Study 03

Supply Chain Logistics Cost & Efficiency Analysis

Tools Power Query · Excel · Power BI
Dataset 209K Order Lines · 7 Tables
Coverage 7 Warehouses · 3 Carriers · 3 Routes
Source Brunel University London
01 — Context

Seven Tables. One Pipeline. A Discontinued Carrier Nobody Noticed.

This dataset from Brunel University London isn't a single clean spreadsheet — it's seven separate files representing a real logistics operation: orders, freight rates, warehouse costs, capacity limits, plant-to-port mappings, and customer routing rules. None of it is pre-joined. None of it is ready to analyse.

The project required building a transformation pipeline in Power Query that merged all seven tables into a single analysis-ready model — then using that model to answer the four operational questions any logistics manager actually needs answered: who to ship with, which warehouse is overloaded, which routes are bleeding money, and where new orders should go.

Business questions answered:

  • —Which carrier has the best cost-to-speed ratio?
  • —Which warehouse is closest to hitting max capacity?
  • —Which routes are consistently over budget?
  • —Where should new orders be routed given current warehouse loads?
  • —How should a discontinued carrier be handled without silently dropping data?
Key Finding

V444_0 Scores 6× Better

V444_1 costs $22.94 per shipment vs V444_0's $3.86 — with only a 2.5 percentage point improvement in on-time rate. The Cost-to-Speed Score ($3.97 vs $23.04) makes the decision clear: V444_0 is the better carrier by every meaningful metric.

Data Integrity Issue

Carrier V44_3 — 854 Orphaned Orders

V44_3 exists in the order history but has no freight rate data — a discontinued carrier mid-operation. Instead of silently dropping the rows, the pipeline flags them with a Carrier_Status column and preserves them for documentation. A real data integrity decision, not a workaround.

02 — Approach

How I Framed the Pipeline

A 3-layer build — Power Query transformation pipeline, Excel pivot analysis, Power BI dashboard — each layer feeding cleanly into the next, with no duplicated logic across tools.

Phase 01

Load & Inspect

All 7 source files loaded as Connection Only queries — raw copies preserved before any transformation. Data types set manually. Join key columns trimmed to prevent invisible whitespace from silently breaking merges downstream.

Phase 02

Merge & Enrich

Five-step Left Outer Join sequence — OrderList as the fact table, enriched with FreightRates, PlantPorts, WarehouseData, ProductsPerPlant, and VmiCustomers. Row count verified after each merge to catch silent join failures.

Phase 03

Calculate & Flag

Freight cost calculated using List.Max to apply the minimum cost floor — replicating real carrier billing logic. V44_3 flagged with a Carrier_Status column. Delivery status, route key, and cost-per-unit columns added before load.

Phase 04

Analyse & Visualise

Excel pivot tables across carrier, warehouse, and route dimensions. Power BI connected directly to MasterOrders — DAX measures for Cost-to-Speed Score and on-time rate. Four-page interactive dashboard with slicers on every page.

03 — Process
01 / Data Loading & Schema Inspection

Seven files. Seven different structures. One pipeline.

Each source file had its own column naming conventions, inconsistent capitalisation, and varying data types. Every join key column — Carrier, Plant Code, Service Level, Origin Port, Destination Port — was trimmed and typed manually before any merge was attempted. Invisible whitespace in join keys is one of the most common and hardest-to-diagnose causes of silent merge failures in Power Query.

Power Query Connection Only Data Type Enforcement Text.Trim
02 / The V44_3 Problem

A discontinued carrier — and a deliberate handling decision

During the FreightRates merge, 854 orders returned null on every rate column — all V44_3 orders. This carrier existed in the order history but had been discontinued, leaving no rate card data. Rather than dropping the rows or imputing values without disclosure, I retained all 854 orders and added a Carrier_Status column flagging them as "Discontinued". The decision is documented, the data is preserved, and every downstream analysis can filter explicitly.

Left Outer Join Null Handling Conditional Column Data Integrity
03 / Freight Cost Calculation

Replicating real carrier billing logic — not just Weight x Rate

Real carriers don't charge a flat per-kg rate — they charge the greater of (weight x rate) or a minimum cost floor. The Freight_Cost column uses List.Max({[minimum cost], [Weight] * [rate]}) to replicate this exactly. For null-rate V44_3 orders, the formula returns null rather than zero — preserving the distinction between "no cost calculated" and "zero cost".

04 / Cost-to-Speed Score & Dashboard

A single metric that makes the carrier decision obvious

Average freight cost alone doesn't tell the full story — a cheap carrier that's always late isn't a bargain. The Cost-to-Speed Score (Avg Cost x (1 + (1 - On-Time Rate))) combines both dimensions into one number, penalising unreliability. Built as a DAX measure in Power BI and reproduced as an Excel formula for cross-validation. V444_0 scores $3.97. V444_1 scores $23.04. The recommendation writes itself.

05 — Key Findings
Carrier & Cost Efficiency

V444_1 costs 6x more. The on-time improvement doesn't justify it.

  • —V444_0 avg cost: $3.86 — V444_1 avg cost: $22.94 per shipment
  • —On-time rate difference: 99.57% vs 97.06% — only 2.5 percentage points
  • —Cost-to-Speed Score: V444_0 = $3.97 — V444_1 = $23.04
  • —V444_1 late shipments avg $30 — even more expensive when it fails
  • —Total freight spend: $2.41M across 209,402 orders at $11.54 avg
Warehouse & Route Risk

94% of orders through one warehouse. 99.9% through one route.

  • —PLANT03: 196,953 of 209,402 orders — single point of failure risk
  • —PLANT08 on-time rate: 82.19% — the only plant below 95%
  • —PORT09 to PORT09: 207 orders at $1,713 avg — 156x the main route cost
  • —PORT04 to PORT09 handles 209,194 orders — a single-lane network
  • —V44_3: 854 orders with no freight cost — preserved and flagged, not dropped

07 — Reflection
Lesson 01

Silent failures are the worst kind

A left join that drops rows doesn't warn you. A whitespace character in a join key doesn't flag itself. The V44_3 discovery only happened because I checked row counts after every merge. Data pipelines fail quietly — building in verification steps after each stage isn't paranoia, it's the job.

Lesson 02

Average cost is the wrong metric for carrier selection

V444_1's $22.94 average vs V444_0's $3.86 looks damning on its own — but it's incomplete. A carrier could be expensive and reliable, or cheap and unreliable. The Cost-to-Speed Score collapses both dimensions into one number and makes the decision unambiguous. Composite metrics almost always tell a clearer story than isolated ones.

Lesson 03

Concentration risk is invisible until you measure it

94% of orders through one warehouse and 99.9% through one route didn't appear in any single column of the source data — it only became visible once the full picture was assembled. Aggregation reveals structure that row-level inspection never could. The most important finding in this project wasn't in the data, it was in what the data looked like from above.

Lesson 04

Anomalies are the most interesting rows

PORT09 to PORT09 at $1,713 average and V44_3 with 854 null freight costs are outliers — the kind of rows most analysts filter away to keep charts clean. But both are telling a story. One is a potential data error worth $116K. The other is an operational gap in the carrier record. The anomalies are often where the most valuable questions live.
